## 第二部分：核心方法论（30分钟，先读这个）

### 5.1 Agent 开发三层次框架

#### 层次 1：基础配置型（基础性工作，适合入门）

**做什么**：
- 开关工具（启用/禁用某些 Tools）
- 修改 System Prompt（改身份描述）
- 配置外部 API 集成

**为什么重要**：
- 理解 Agent 运行机制的第一步
- 产品经理/运维就能做，但工程师需要理解底层原理
- 为后续层次打基础

---

#### 层次 2：工具开发型（需要领域知识才能做对）

**做什么**：
- 封装外部 API 为 Tool
- 简单数据处理工具
- 文件操作增强

**风险与机会**：
- 简单工具封装可能被 LLM 自动生成（未来趋势）
- 标准化工具（如查天气、搜网页）已由框架提供
- **护城河**：复杂业务逻辑（需要对领域有深度理解）、多步骤编排（非单一功能）

---

#### 层次 3：系统设计型（⭐ 核心目标，高价值，难以被取代）

**做什么**：
1. **垂直领域深度封装**
   - 不是"查股价"，而是"完整回测流程"
   - 不是"读文件"，而是"策略风险扫描"

2. **复杂工作流编排**
   - 多 Agent 协作设计
   - 状态流转和错误回退机制
   - 领域专属的流程编排

3. **领域知识系统化沉淀**
   - 将专家经验结构化为 Skill
   - 场景模板、最佳实践、常见错误

**为什么不可替代**：
- 需要深度领域知识（量化交易、风控、合规）
- 需要系统设计能力（架构、流程、状态管理）
- 需要工程落地能力（不是纸上谈兵）

---

#### 核心目标

> **掌握高层次的 Agent 系统设计能力，能够针对垂直领域设计并实现专业级的 Agent 系统。**

本方法论将通过 **Hermes Agent（学习）+ Claude Code（参考）+ CodeStudio（实践）+ StarQuant（实践）** 四个项目具体实践验证。

> ⚠️ **Claude Code 使用说明**：Claude Code 仅用于**代码分析**（只读仓库、对比架构设计），**不编译、不启动**。如需深入某模块，直接阅读对应 `.ts` 文件即可。

### 5.2 学习方法论：问题驱动 + 产出物倒逼的螺旋

> 从 Claude Code 仓库的 `mydocs/00-1-学习路径规划.md` 提取的先进学习方法。

#### 不采用的路径
先学完再反推项目（记忆衰减快、噪音信号难分辨、学完不用就忘）

#### 采用的路径

```
每个子阶段固定 5 步（无论哪个大阶段都一样）：

  Step 1: 明确问题    ← 本子阶段要解决 Vacky 的什么具体痛点
  Step 2: 现状盘点    ← 看 Vacky 自己的项目，激活问题意识
  Step 3: 定向学习    ← 带着问题去看 Hermes Agent/Claude Code 的相关设计
  Step 4: 产出设计    ← 写出能落到 Vacky 项目的设计文档/图
  Step 5: 自验证      ← 按 Checklist 确认目标达成
```

### 5.3 Vacky 的学习偏好（固定约束）

| 偏好 | 含义 | 反面（不要做的） |
|---|---|---|
| **要设计不要代码** | 讲流程图、状态机、决策树、伪代码 | 贴 TypeScript 源码、讲类型签名 |
| **要类比不要术语** | 用游戏后台（Java/Kotlin 后台）的概念类比 | 直接用 Agent/LLM 术语不解释 |
| **要流程不要清单** | 有先后顺序、因果关系的叙述 | 扁平化的 "知识点列表" |
| **要取舍不要全讲** | 明确告诉哪个重要、哪个次要、哪个跳过 | "这些都了解一下" |
| **要产出不要感悟** | 每个子阶段必须有文件产出 | "你自己体会一下" |

### 5.4 子阶段固定模板（所有子阶段都按此模板）

每个子阶段文档必须包含以下 **10 个固定 section**：

```
### 子阶段 X.Y：<主题名>

【服务项目】CodeStudio / StarQuant / Hermes Agent / Claude Code / 通用
【建议时长】N 天（日历天数，业余学习按每天 1-2 小时计算）
【前置】X.(Y-1) 已完成
【推荐模型等级】* / ** / ***

─────────────────────────────
1. 问题定义
   本子阶段要回答 Vacky 的哪个具体痛点？
   （一句话，必须和真实项目的真实痛点挂钩）

2. 最小知识包
   本子阶段涉及的 3-5 个核心概念，每个一句话定义。
   AI 讲解时必须先对齐这些定义，避免术语漂移。

3. 项目现状盘点（Vacky 的 Day 1-2）
   Vacky 要先做什么动作来审视自己的项目？
   （明确到可执行的动作清单，业余时间可完成）

4. 定向学习内容（Vacky + AI 的 Day 3-4）
   AI 该给 Vacky 讲什么？
   （明确到"该画哪张图、讲哪几个概念、用游戏后台的什么类比"）

5. 产出物（Vacky 的 Day 5-6）
   Vacky 要产出什么具体文件？
   （明确到文件名、文件结构）

6. 自验证清单
   怎么判断目标达成？
   （明确到"能回答 X 问、能画出 Y 图"的可检查标准）

7. AI 准入 3 问
   新 AI 接手时 Vacky 拿这 3 问来"面试"AI。
   AI 答不上这 3 问说明它没真的看懂本子阶段，不要让它继续。

8. 求助模板
   Vacky 卡住时发给 AI 的标准问题。

9. 记录区
   Vacky 完成后填写的心得、发现
   （迭代优化这份手册）

10. 下一步
   自动进入哪个子阶段
```

## 第五部分：设计原则（按需查阅，方法论的深化）

### 5.1 LLM 失败模式（必须掌握）

> 本小节回答学习目标 1："理解模型的工作机制、能力边界与失败场景"

#### 六种核心失败模式

| 失败模式 | 表现 | 在 Agent 设计中的意义 | Hermes Agent 应对 |
|---------|------|---------------------|---------------------|
| **Hallucination** 幻觉 | 编造工具调用参数、返回不存在的数据 | 必须设计 validate → retry 机制 | `Tool.ts` 的 `validateInput()` + `checkPermissions()` |
| **Context Poisoning** 上下文污染 | 早期错误被后续推理放大 | 不能无限制保留历史消息 | `services/compact/` 多策略并存 |
| **Tool Selection Error** 工具选择错误 | 选错工具或参数 | 工具 schema 设计是核心能力 | `Tool.ts` 的 `description()` 动态生成 |
| **Chain-of-Thought Drift** 推理链漂移 | 长推理链中偏离原始目标 | 多步骤工作流必须有人工检查点 | Hooks 系统 `PostToolUse` 验证 |
| **Token 累积误差** | 长对话中 cost 指数增长 | 系统设计时必须考虑成本 | `utils/toolResultStorage.ts` 超限落盘 |
| **Overconfidence** 过度自信 | 模型返回错误结果但不报错 | 关键操作必须人工确认 | 权限系统 `mode: 'ask'` |

---

#### Hermes Agent 的防御体系（6 层）

```
Layer 1: Tool 层 → validateInput() 参数校验
Layer 2: 权限层 → checkPermissions() 权限检查
Layer 3: Hooks 层 → PreToolUse/PostToolUse 拦截
Layer 4: 预算层 → iteration_budget 成本控制
Layer 5: 压缩层 → autoCompact 上下文压缩
Layer 6: 人工层 → 高风险操作人工确认
```

### 5.2 Agent 适用性判断

> 本小节回答："什么时候用 Agent，什么时候不用"

#### 决策树

```
任务是否需要创造性判断？
├─ 否 → 规则系统（if-else）就够用
│   └─ 例：定时拉取数据 → Cron + 脚本
├─ 是 → 是否需要高精度计算？
    ├─ 是 → 调用专业库（numpy、quantlib），Agent 只做编排
    │   └─ 例：期权定价 → Agent 调用 quantlib，不自己算
    └─ 否 → 是否需要处理非结构化输入？
        ├─ 是 → ✅ 适合 Agent（LLM 的泛化能力有价值）
        │   └─ 例：根据代码变更决定重构策略
        └─ 否 → 是否可以枚举所有情况？
            ├─ 是 → 状态机 + 查表
            └─ 否 → ✅ 适合 Agent
```

---

#### 场景对照表

| 场景 | 是否适合 Agent | 理由 | Hermes Agent 对应 | 应用项目 |
|------|----------------|------|---------------------|----------|
| 固定流程（定时拉取数据） | ❌ 不适合 | 规则系统更简单、可预测 | `cron/` + 脚本 | CodeStudio |
| 需要创造性（代码变更决定重构） | ✅ 适合 | LLM 的泛化能力有价值 | `tools/code_tools.py` | CodeStudio |
| 高精度计算（期权定价） | ❌ 不适合 | LLM 数值计算不可靠 | 调用 `quantlib` | StarQuant |
| 需要领域判断（策略回测是否过拟合） | ✅ 适合 | 需要综合多个指标做判断 | `tools/analysis_tools.py` | StarQuant |
| 实时性要求高（高频交易） | ❌ 不适合 | LLM 延迟太高 | 规则系统 + C++ | StarQuant |
| 多步骤研究任务 | ✅ 适合 | 需要灵活调整研究路径 | `tools/research_tools.py` | StarQuant |

### 6.2 工具设计 Checklist（15 维度）

> 本 Checklist 来自 Claude Code 的 Tool 设计实践，适用于所有 Agent 项目的 Tool 开发。

#### Tool 合约的 4 层

| 层次 | 内容 | 示例 |
|------|------|------|
| **Layer 1: Schema 层** | 工具的名称、描述、参数定义 | `name`, `description`, `input_schema` |
| **Layer 2: 行为层** | 工具的实际执行逻辑 | `execute()`, `validate()`, `handle_error()` |
| **Layer 3: 权限层** | 工具的访问控制规则 | `checkPermissions()`, `mode: 'ask'/'auto'/'deny'` |
| **Layer 4: 元信息层** | 工具的分类、标签、使用统计 | `category`, `tags`, `usage_count` |

---

#### 15 维度 Checklist

1. **命名清晰度**：工具名称是否能让 LLM 一眼看懂用途？
2. **描述完整性**：`description` 是否包含：功能、参数、返回值、失败场景？
3. **参数必要性**：每个参数是否都是必需的？能否合并或删除？
4. **参数默认值**：是否有合理的默认值？是否会导致意外行为？
5. **输入校验**：是否检查了参数类型、范围、格式？
6. **错误返回**：失败时是否返回结构化错误（而非裸异常）？
7. **权限控制**：是否需要人工确认？哪些场景需要？
8. **超时处理**：工具执行超时怎么办？是否有重试机制？
9. **幂等性**：重复调用是否安全？是否需要幂等性保证？
10. **副作用**：工具是否有副作用（写文件、发请求）？是否明确告知 LLM？
11. **可组合性**：工具能否和其他工具组合使用？数据流转是否清晰？
12. **性能保证**：工具执行时间是否可控？是否需要异步？
13. **日志记录**：关键操作是否记录了日志？便于调试和审计？
14. **版本兼容**：工具接口变更是否向后兼容？是否需要版本号？
15. **文档覆盖**：是否有使用示例？是否覆盖了常见场景和边界 case？

---

### 6.3 架构设计灵魂问题（10 问）

> 本 10 问用于检验 Agent 系统架构设计是否完整，适用于阶段 4 的架构设计验证。

1. **主循环边界**：Agent 的主循环在哪里结束？什么条件下退出？
2. **工具调度策略**：多个工具调用请求，如何决定调用顺序？并发还是串行？
3. **错误处理边界**：工具执行失败，是重试、跳过、还是终止整个任务？
4. **上下文管理策略**：长对话如何压缩？压缩后如何保证关键信息不丢失？
5. **权限控制粒度**：哪些操作需要人工确认？确认内容是什么？
6. **状态持久化**：Agent 的中间状态在哪里保存？如何恢复？
7. **多 Agent 协作**：如果是多 Agent 系统，Agent 之间如何通信？如何避免冲突？
8. **成本可控性**：如何防止 Agent 无限调用 API？预算耗尽后如何处理？
9. **可观测性**：如何追踪 Agent 的每一步操作？如何调试？
10. **领域适配性**：架构是否适配当前领域？是否需要领域专属的优化？
