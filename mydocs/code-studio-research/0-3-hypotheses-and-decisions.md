## 二、设计决策记录（DR）

> 记录关键设计决策和推理过程。

### DR-001：是否实现形式化 Gates？

**决策**：✅ 是，要实现形式化 Gates（相比 CCGS 的软性 Gates）

**理由**：

1. CCGS 的 Gates 依赖 Agent 自愿遵守，没有强制机制

2. 生产环境的流程确认必须有强制性（否则没有意义）

3. 形式化验证（JSON Schema）可以自动化，减少人工审查成本

**替代方案**：

- ❌ 软性 Gates（像 CCGS 那样）→ 不安全
- ❌ 完全人工审查 → 成本高

**状态**：✅ 已决策，待设计

---

### DR-002：知识系统采用混合模式（静态 + 动态）

**决策**：✅ 是，采用混合模式（静态知识 + 动态知识）

**理由**：

1. 静态知识（项目架构、部署流程）稳定，不常变，适合手动编写

2. 动态知识（常见 bug、调试经验）易变，需从实践中持续更新

3. 混合模式兼顾简单性和进化能力

**替代方案**：

- ❌ 纯静态知识 → 不会进化，可能过时
- ❌ 纯动态知识 → 复杂，需要设计提炼算法

**状态**：✅ 已决策，待设计 Evidence Chain

---

### DR-003：安全机制采用 Sandbox + Checkpoint

**决策**：✅ 是，实现 Sandbox + Checkpoint + Audit Log

**理由**：

1. 服务器开发高风险，必须有安全机制

2. Sandbox 可以在隔离环境预演操作

3. Checkpoint 提供回滚能力

4. Audit Log 记录所有操作，便于追责

**替代方案**：

- ❌ 无安全机制 → 高风险
- ❌ 只有 Dry-run → 无法回滚

**状态**：✅ 已决策，待设计 Safety Module

---

### DR-004：知识全生命周期管理策略

**决策**：✅ 是，实现知识全生命周期管理（初始化、组织、注入、更新、废弃）

**理由**：

1. 知识对 CodeStudio 目标实现非常重要，但当前无系统化管理

2. 知识的价值随时间变化（初始化 → 成熟 → 过时 → 废弃）

3. 需要系统化的管理策略，避免知识库膨胀、过期、冲突

**核心设计**：

1. **初始化策略**：三阶段初始化（人工编写 → 自动提取 → 实践提炼）

2. **组织策略**：知识分类（静态/动态/流程/上下文）+ 知识索引（关键词 + 语义搜索）

3. **注入策略**：智能注入（根据触发时机、注入内容、注入量）

4. **更新策略**：Evidence Chain 驱动更新（检测过时 → 标记可疑 → 人工审核 → 更新）

5. **废弃策略**：安全废弃（检测废弃条件 → 标记废弃 → 观察期 → 永久删除）

**替代方案**：

- ❌ 无知识管理 → 知识库混乱，效率低下
- ❌ 只有静态知识 → 不会进化，可能过时
- ❌ 只有动态知识 → 冷启动问题，质量不稳定

**状态**：✅ 已决策，待设计 Knowledge Lifecycle Manager

---

### DR-005：采用 Harness Engineering 模式

**决策**：✅ 是，采用 Harness Engineering 模式（而非直接让 AI 写代码）

**理由**：

1. 直接让 AI 写代码，缺乏规范、约束、验证，容易出错

2. Harness Engineering 提供"工程化框架"，AI 在框架内工作，强制规范、约束、验证

3. 类比：不是让 AI 直接开车，而是构建自动驾驶系统（有轨道、有规则、有验证）

**核心设计**：

1. **规范层**（GD-1）：用形式化语言定义"应该做什么"（JSON Schema）

2. **约束层**（GD-2）：将规范实例化为具体项目的约束

3. **验证层**（Enforcer）：在关键节点自动验证约束（Pre-tool Hook）

4. **反馈层**（Evidence Chain）：收集操作证据，用于知识更新和流程改进

**替代方案**：

- ❌ 直接让 AI 写代码 → 无规范、无约束、无验证
- ❌ 只有软性规范（像 CCGS）→ Agent 可以不遵守

**状态**：✅ 已决策，待设计 Harness Framework

---

### DR-006：系统思维注入机制

**决策**：✅ 是，实现系统思维注入（解决 AI 的局部性问题）

**理由**：

1. 我们在实践中发现，AI 在问题排查、修复时具有严重的局部性

2. 局部性表现为：缺乏全局依赖分析、缺乏深层根因分析、缺乏架构演进思维

3. 需要系统化的注入机制，让 AI 具备系统思维

**核心设计**：

1. **系统视图注入**：修改代码前，先注入系统架构图、依赖图、影响面分析

2. **根因分析框架注入**：修复 bug 前，强制完成根因分析（5 个问题）

3. **架构原则注入**：做架构决策时，注入架构原则（复用优先、底层优化优先、系统思考、Tech Debt 管理）

4. **系统思维训练**：通过实践，训练 AI 的系统思维（案例库 + 反思机制 + 人类反馈）

**替代方案**：

- ❌ 无系统思维注入 → AI 只做局部优化，不做系统思考
- ❌ 只有一次性注入 → 无法持续进化

**状态**：✅ 已决策，待设计 System Thinking Injector

---

### DR-007：CodeStudio 能力金字塔

**决策**：✅ 是，定义 CodeStudio 能力金字塔（而非只做代码生成）

**理由**：

1. CodeStudio 不是"AI 代码生成器"，而是"AI 系统工程助手"

2. 需要明确能力层级，避免只关注代码生成（低层），忽略系统设计、系统优化、系统演进（高层）

3. 能力金字塔：代码生成 → 系统设计 → 系统优化 → 系统演进

**核心设计**：

1. **第一层（基础）**：代码生成、Bug 修复

2. **第二层**：架构设计、模块设计

3. **第三层**：性能优化、稳定性优化

4. **最高层**：架构演进、技术债管理

**实现策略**：

1. **技术债管理**：自动检测 `HACK`、`TEMP`、`WORK_AROUND`，自动创建 Tech Debt 卡片

2. **架构演进规划**：自动生成架构图，对比"当前架构"和"目标架构"

3. **系统思考训练**：系统思维案例库 + 反思机制

**替代方案**：

- ❌ 只做代码生成 → 无法解决系统级问题
- ❌ 没有能力层级 → 无法系统化提升 AI 能力

**状态**：✅ 已决策，待设计 Capability Pyramid

#### DR-008：知识分类框架（基于知识本质分类）

**决策**：✅ 是，按知识本质分类，不同类型采用不同存储和触发方式

**理由**：

1. 知识不是均质的，不同类型的知识需要不同的处理方式
2. 程序性知识（固定流程）应该封装为 Skill（代码），而非文档
3. 上下文知识（需要理解系统设计）应该存为 Knowledge（文档），而非 Skill
4. 架构性知识（系统结构）应该在会话启动时注入
5. 经验性知识（从实践中学习）应该通过 Evidence Chain 管理

**核心设计**：

| 知识类型 | 本质 | 示例 | 应该存为 | 触发方式 |
|---------|------|------|----------|----------|
| **程序性知识**（Procedural） | 固定流程、可自动化 | 编译、重启、部署 | ✅ **Skill**（代码） | 任务触发 |
| **上下文知识**（Contextual） | 任务背景、设计思想 | 如何新增活动类型 | ✅ **Knowledge**（文档） | 上下文注入 |
| **架构性知识**（Architectural） | 系统结构、模块关系 | 项目架构图、依赖关系 | ✅ **Knowledge**（文档） | 会话启动注入 |
| **经验性知识**（Empirical） | 从实践中学习 | 常见 bug、解决方案 | ✅ **Evidence Chain** | 错误时注入 |

**示例**：

1. **编译和重启 letsgo_server** → 程序性知识 → 封装为 Skill
2. **新增活动类型** → 上下文知识 → 存为 Knowledge 文档

**替代方案**：

- ❌ 所有知识都存为文档 → AI 无法直接执行固定流程
- ❌ 所有知识都封装为 Skill → 上下文知识不适合自动化

**待解决问题**：

1. ❓ 如何自动识别"程序性知识"和"上下文知识"？（避免人工分类）
2. ❓ 当知识类型变化时，如何迁移？（Knowledge → Skill）
3. ❓ 如何保证 Skill 和 Knowledge 的一致性？

**状态**：📝 已决策（初步），待实践验证
