# s1 学习讨论 — Agent 基础认知

> 本文件记录 s1（基础认知阶段）学习过程中的讨论、问答和背景梳理，帮助建立整体认知后再深入代码。

---

## 讨论一：Agent 能做什么？为什么需要 Agent 开发？它和前后台开发有什么区别？

### Agent 能做什么？

Agent = **能自主使用工具 + 多轮推理 + 完成开放式任务** 的程序。

| 场景 | 具体能力 |
|------|----------|
| 代码助手 | 读懂仓库、改代码、跑测试、修 Bug |
| 自动化办公 | 读邮件 → 查数据库 → 生成报告 → 发通知 |
| 数据分析 | 下载数据 → 清洗 → 可视化 → 写结论 |
| 运维辅助 | 看告警 → 查日志 → 定位根因 → 给出修复建议 |

**传统程序**：输入 → 固定逻辑 → 输出（确定性）

**Agent**：目标 → 自主规划 → 调用工具 → 观察结果 → 调整策略 → 最终交付

---

### 为什么需要 Agent 开发？

传统后台开发的瓶颈：你要把所有逻辑写死。

```
用户说："帮我分析上周的异常订单，找出原因，发邮件给供应链"
```

- **传统做法**：产品写 PRD → 你写接口 → 写逻辑 → 测试 → 上线（几天到几周）
- **Agent 做法**：直接把这句话给 Agent，它自己拆解任务、调工具、完成（秒级）

根本驱动力：
- LLM 提供了**理解自然语言 + 推理 + 生成**的能力
- 但 LLM 本身**不能执行操作**（不能读文件、不能调 API、不能跑代码）
- **Agent 开发 = 把 LLM 的"大脑"和"手脚"（工具）连接起来**

---

### Agent 开发 vs 前后台开发

核心区别一句话：

> **前后台开发是"写固定逻辑"，Agent 开发是"教 LLM 如何使用工具自主决策"。**

| 维度 | 后台开发 | 前台开发 | Agent 开发 |
|------|----------|----------|------------|
| 核心工作 | 写业务逻辑、CRUD、API | 写 UI、交互、状态管理 | 设计工具、Prompt、编排逻辑 |
| 确定性 | 高，输入输出固定 | 高，用户操作响应固定 | **低，同输入可能不同输出** |
| 调试方式 | 单步调试、单测 | DevTools、日志 | **看 LLM 思考过程、评估质量** |
| 失败模式 | 报错、异常 | UI bug、状态不一致 | **幻觉、工具调用错误、循环** |
| 主要挑战 | 并发、一致性、性能 | 兼容性、体验 | **可靠性、成本控制、上下文管理** |

**用后台经验类比**：

传统后台像写游戏规则，所有分支你都写死。

Agent 开发像设计一个 NPC 的"智能"：你不直接控制 NPC 每一步做什么，你给它**工具**（捡东西、开门、攻击），给它**目标**（"去拿宝箱"），它自己决定怎么走。区别在于：NPC 的行为是**概率性**的，这也是 Agent 开发最头疼的地方。

## 讨论二：Coding Agent 为什么有落地空间？稳定性不是最重要的吗？

这个判断非常准确，抓住了当前 Agent 落地的核心矛盾。

### 稳定性是线上服务的生命线

Agent 目前的几个硬伤：

- **非确定性**：同样输入，输出可能不同
- **幻觉**：可能"自信地"返回错误结果
- **不可解释**：很难 100% 说清楚它为什么这样做

这几个问题放在面向用户的线上服务里，基本是**不可接受的**。

---

### 为什么 Coding Agent 有空间？

关键在于 **Human-in-the-loop（人在回路中）**：

```
传统线上服务：  用户请求 → 服务处理 → 返回结果（必须正确）
Coding Agent：  开发者指令 → Agent 操作 → 开发者审查 → 接受/拒绝
```

| 维度 | 面向用户的 Agent | Coding Agent |
|------|-----------------|-------------|
| 错误成本 | 高，直接影响用户 | 低，开发者可以 review |
| 验证方式 | 很难自动验证 | **代码可以跑测试** |
| 容错性 | 几乎为零 | 相对较高 |
| 回滚成本 | 高 | 低（git revert） |

代码有一个巨大优势：**可验证性**。Agent 改完代码，跑一下测试就知道对不对。这是目前 Agent 能落地的少数场景之一。

---

### 延伸：什么时候 Agent 能直接服务用户？

需要解决几个工程问题：

1. **确定性执行**：Planning 阶段和执行阶段分离，先生成"执行计划"，用户确认后再执行
2. **可验证的输出**：输出必须有结构化校验（JSON Schema、类型检查），关键操作需要 dry-run 机制
3. **降级策略**：Agent 不确定时主动说"我不知道"，重要操作必须走人工审批

---

## 讨论三（高级工程师视角）：客观评估 Agent 的价值、短板和开发内容

### 已验证的实际价值

1. **大幅降低长尾任务开发成本**：企业内部跨系统的自动化需求，开发周期可以压缩 70% 以上
2. **显著提升开发者效率**：Coding Agent 可以帮助开发者减少 30%~50% 的重复性开发工作
3. **降低非技术人员的使用门槛**：运营人员可以通过自然语言直接提取数据、生成报告

---

### 当前无法回避的核心短板

1. **稳定性不足**：非确定性导致输出可能出错，当前没有 100% 的解决手段，只能靠工程手段缓解
2. **成本高、延迟高**：复杂任务可能需要调用几十次大模型，成本和延迟都无法满足高并发、低延迟场景
3. **可解释性差**：很难 100% 回溯 Agent 做某个决策的原因
4. **能力上限受限于工具丰富度**：Agent 做不到工具覆盖范围以外的能力

---

### Agent 开发的核心工作内容

当前 Agent 开发还处于**工程化落地早期**阶段，核心工作围绕"让非确定性的大模型系统满足生产级稳定性要求"展开：

| 工作方向 | 具体内容 | 后台经验复用度 |
|----------|----------|----------------|
| 工具工程 | 设计工具 Schema、鉴权、限流、降级 | ⭐⭐⭐⭐⭐ 完全复用 |
| 可靠性工程 | 迭代预算、高风险审批、失败重试、轨迹回放 | ⭐⭐⭐⭐⭐ 完全复用 |
| Prompt 工程 | 设计系统提示词、定义行为边界 | ⭐⭐ 新方法，需学习 |
| 编排逻辑开发 | 实现 Agent 主循环、处理边界情况 | ⭐⭐⭐⭐ 逻辑类似服务编排 |
| 上下文工程 | 长对话压缩、记忆召回、防止溢出 | ⭐⭐⭐ 新领域，有借鉴 |

**结论**：有后台开发经验的人在**可靠性工程、工具设计**上有天然优势，这也是 Agent 能否上生产的关键。

---

## 小结

- Agent 不是替代前后台开发，而是补位**长尾、非结构化、跨系统协同**的场景
- 当前最成熟的落地场景是 **Coding Agent**，核心是"可验证 + 人在回路"
- Agent 开发的核心挑战不是"让模型更聪明"，而是**用工程手段让非确定性系统变得可控**
- 后台工程师转型 Agent 开发，在可靠性、工具设计上有天然优势

## 讨论四：从 Claude Code Game Studios 看 Agent 到底能做什么？

### 背景

问题出发点是：

> 如果 Agent 不能提供一个稳定上线对外服务的后台服务器、客户端或策划案，那 LLM/Agent 具体能做什么？

为了回答这个问题，我们阅读了 [Claude Code Game Studios](https://github.com/Donchitos/Claude-Code-Game-Studios) 这个仓库。

---

### CCGS 做了什么？

**一句话概括**：把一个完整的游戏工作室流程，结构化地"装进" Claude Code 里。

它不是让 AI 自主开发游戏，而是把游戏开发的所有环节——从脑暴、设计文档、架构决策、代码实现、QA——都变成了**有结构的协作流程**。

#### 核心数据

| 组件 | 数量 | 作用 |
|------|------|------|
| 专用 Agent | 49 个 | 分三层：总监（Opus）→ 部门负责人（Sonnet）→ 专员（Sonnet/Haiku） |
| 技能（Slash 命令） | 72 个 | 覆盖从脑暴到发布的完整工作流 |
| 自动化 Hook | 12 个 | 提交检查、资源规范检查、会话审计等 |
| 路径级规则 | 11 条 | 不同目录（gameplay、network、ui...）强制执行不同的编码规范 |

#### 工作流程（7 个阶段）

```
Concept → Systems Design → Technical Setup → Pre-Production → Production → Polish → Release
```

每个阶段有多个步骤，每个步骤对应一个 `/slash` 命令，由专门的 Agent 执行。

---

### 关键设计原则：Collaborative, Not Autonomous

这是回答"Agent 能做什么"的核心。CCGS 的 README 明确写了：

> **You stay in control. The agents provide structure and expertise, not autonomy.**

所有 Agent 严格遵循**五步协作协议**：

```
1. Ask        → Agent 在提出方案前先问问题
2. Present    → 展示 2-4 个方案，附 pros/cons
3. You decide → 用户做最终决策
4. Draft      → Agent 展示工作成果 before 定稿
5. Approve    → 没有你的签字，什么都不写
```

---

### 回答原问题：Agent 在游戏侧（以及后台开发）到底能做什么？

#### ✅ 能做的（且做得好）

| 能做的 | 具体表现 | 为什么适合 Agent |
|--------|----------|-----------------|
| **结构化流程引导** | 把游戏开发拆成 7 个阶段、几十个步骤，告诉你现在该做什么、下一步是什么 | LLM 擅长理解和执行流程 |
| **文档起草** | 生成 GDD、ADR、UX 规格、QA 计划等文档的初稿 | LLM 擅长生成结构化文本 |
| **方案对比** | 给你 2-4 个方案，列出 pros/cons，帮你决策 | LLM 擅长多角度分析 |
| **代码 Review** | 按照 coding standards 检查代码，提出修改建议 | 规则明确，可验证 |
| **自动化检查** | Hook 自动检查 hardcode、命名规范、JSON 格式等 | 确定性的检查用脚本做 |
| **知识检索与应用** | 应用 MDA 框架、Self-Determination Theory 等游戏设计理论 | LLM 擅长应用已有知识 |

#### ❌ 不能做的（当前技术限制）

| 不能做的 | 原因 |
|----------|------|
| **自主完成一个可上线的系统** | 非确定性，没有人的审查和测试不行 |
| **保证代码没有 Bug** | 幻觉问题，可能"自信地"写出有问题的代码 |
| **替代人的创意决策** | 设计方向、玩法取舍，必须由人决定 |
| **处理复杂的线上问题** | 缺乏真正的理解和调试能力 |

---

### 对游戏后台开发的具体价值

你说得对，**稳定性是生命线**。Agent 不能直接给你一个可上线的后台服务。

但它能做的很有价值，用工程语言描述就是：

```
传统做法：
  你：写设计文档 → 画架构图 → 写代码 → 写单测 → 做 code review
  时间：几天到几周

Agent 辅助做法：
  Agent：帮你起草设计文档初稿 → 生成架构方案对比 → 生成代码骨架 → 生成单测模板 → 做初步 review
  你：审查、决策、修改、最终确认
  时间：几小时到几天
```

**核心价值 = 放大你的生产力，而不是替代你。**

具体到后台开发场景：

1. **架构设计**：Agent 可以帮你生成几个候选架构方案，列出 pros/cons，你来做最终决策
2. **代码实现**：Agent 可以写初版代码，你来 review、测试、迭代
3. **文档编写**：Agent 可以帮你写设计文档、API 文档、运维手册的初稿
4. **测试生成**：Agent 可以帮你生成单测、集成测试的模板
5. **Code Review 辅助**：Agent 可以帮你做第一遍 review，筛选出明显问题

---

### 小结

- Agent 的定位是**协作伙伴，不是自主工作者**
- 它的核心价值在**起草、结构化、对比分析、初步审查**，最终决策权在人手里
- 对后台开发来说，Agent 是**生产力放大器**，不是替代者
- 当前最合理的用法是 **Human-in-the-loop**，这也是 Coding Agent 能落地的根本原因
