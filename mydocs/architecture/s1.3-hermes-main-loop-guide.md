## 步骤1：理解泳道图（预计15分钟）

### 1.1 什么是泳道图？

泳道图（Swimlane Diagram）是一种流程图，将活动组织到不同的"泳道"中，每个泳道代表一个责任区域或参与者。

**泳道图的组成**：
- **泳道**：横向或纵向的区域，代表不同的参与者或系统组件
- **活动**：泳道中的步骤或操作
- **流程线**：连接活动的箭头，表示流程方向

### 1.2 为什么需要泳道图？

1. **可视化复杂流程**：将复杂的代码逻辑转化为直观的图形
2. **识别责任边界**：清楚看到每个组件的职责
3. **发现优化点**：通过可视化，容易发现流程中的瓶颈或冗余

### 1.3 Hermes Agent 主循环的泳道设计

根据 `s2.1-agent-loop.md` 的分析，Hermes Agent 主循环可以分为以下泳道：

1. **用户**：发起请求，接收回复
2. **CLI**：处理用户输入，显示输出
3. **AIAgent**：核心循环控制器
4. **LLM API**：语言模型服务
5. **工具系统**：执行工具调用
6. **上下文管理**：管理对话历史和压缩

**提示**：你可能需要根据实际代码调整泳道数量。

## 步骤3：分析 Hermes Agent 主循环（预计1-2小时）

### 3.1 阅读关键代码

**文件1：`run_agent.py`**
- 重点阅读：`run_conversation()` 方法
- 关注点：循环条件、API调用、工具调用处理

**文件2：`model_tools.py`**
- 重点阅读：`handle_function_call()` 函数
- 关注点：工具调度、执行、结果返回

**文件3：`tools/registry.py`**
- 重点阅读：`register()` 方法
- 关注点：工具注册、发现

### 3.2 识别关键步骤

根据 `s2.1-agent-loop.md` 的分析，主循环的关键步骤包括：

1. **初始化**：
   - 生成 task_id
   - 重置重试计数器
   - 设置 iteration_budget
   - 加载/构建 system_prompt
   - 预压缩上下文（如果超出阈值）

2. **添加用户消息**：
   - 将用户输入添加到 messages[]

3. **循环开始**：
   - 检查中断请求
   - 检查预算（iteration_budget）
   - 构建 api_messages（注入 context、清理字段）
   - 调用 LLM API（带重试机制）

4. **处理响应**：
   - 被截断（length）→ 请求继续/重试
   - 有 tool_calls → 执行工具 → 结果加入 messages → continue
   - 纯文本 → 作为 final_response → break

5. **错误恢复**：
   - 压缩上下文（413/上下文溢出）
   - 切换 fallback provider（429/401）
   - 指数退避重试（其他错误）

6. **收尾**：
   - 保存 session
   - 触发 post_llm_call hook
   - 后台 memory/skill review
   - 返回结果 dict

### 3.3 分配到泳道

将关键步骤分配到对应的泳道：

**泳道1：用户**
- 输入请求
- 接收回复

**泳道2：CLI**
- 显示输入提示
- 显示输出
- 处理中断信号

**泳道3：AIAgent**
- 初始化
- 添加用户消息
- 循环控制
- 构建 api_messages
- 处理响应
- 错误恢复
- 收尾

**泳道4：LLM API**
- 接收请求
- 生成回复
- 返回 tool_calls 或 content

**泳道5：工具系统**
- 执行工具
- 返回结果

**泳道6：上下文管理**
- 压缩上下文
- 管理对话历史

**提示**：
- 这个分配可能不是最优的，你可以根据实际代码调整
- 重点是理解流程，而不是完美分配

1. **初始化时**：如果上下文超出阈值，预压缩

### 4.4 示例流程

```
用户 → CLI → AIAgent → LLM API → AIAgent → CLI → 用户

```
用户 → CLI → AIAgent → LLM API → AIAgent → 工具系统 → AIAgent → LLM API → AIAgent → CLI → 用户
