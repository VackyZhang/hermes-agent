## 1. Tool 合约的 4 层

### Layer 1: Schema 层
- **内容**：工具的名称、描述、参数定义
- **Claude Code 实现**：`Tool.ts` 的 `inputSchema: z.ZodSchema`
- **Hermes Agent 对应**：`tools/*.py` 的 `schema=` 参数

### Layer 2: 行为层
- **内容**：工具的实际执行逻辑
- **Claude Code 实现**：`Tool.ts` 的 `execute()` 方法
- **Hermes Agent 对应**：`handler=` 参数（lambda 函数）

### Layer 3: 权限层
- **内容**：工具的访问控制规则
- **Claude Code 实现**：`Tool.ts` 的 `checkPermissions()` 方法
- **Hermes Agent 对应**：`check_fn=` 参数

### Layer 4: 元信息层
- **内容**：工具的分类、标签、使用统计
- **Claude Code 实现**：`Tool.ts` 的 `category`, `tags` 属性
- **Hermes Agent 对应**：`tools/registry.py` 的 `category=` 参数

## 2. 15 维度 Checklist

> 为自己的项目（StarQuant）设计 Tool 时，按此 Checklist 逐项检查。

1. **命名清晰度**：工具名称是否能让 LLM 一眼看懂用途？
2. **描述完整性**：`description` 是否包含：功能、参数、返回值、失败场景？
3. **参数必要性**：每个参数是否都是必需的？能否合并或删除？
4. **参数默认值**：是否有合理的默认值？是否会导致意外行为？
5. **输入校验**：是否检查了参数类型、范围、格式？
6. **错误返回**：失败时是否返回结构化错误（而非裸异常）？
7. **权限控制**：是否需要人工确认？哪些场景需要？
8. **超时处理**：工具执行超时怎么办？是否有重试机制？
9. **幂等性**：重复调用是否安全？是否需要幂等性保证？
10. **副作用**：工具是否有副作用（写文件、发请求）？是否明确告知 LLM？
11. **可组合性**：工具能否和其他工具组合使用？数据流转是否清晰？
12. **性能保证**：工具执行时间是否可控？是否需要异步？
13. **日志记录**：关键操作是否记录了日志？便于调试和审计？
14. **版本兼容**：工具接口变更是否向后兼容？是否需要版本号？
15. **文档覆盖**：是否有使用示例？是否覆盖了常见场景和边界 case？
